# 🚨 CogniDispatch — AI-Powered Emergency Home Service Triage

CogniDispatch is a next-generation, real-time emergency dispatch platform for home services (Plumbing, Electrical, HVAC, Structural). It uses **Azure Cognitive Services** to allow homeowners to report emergencies via voice, uses **AI Triage** to classify the emergency, and instantly dispatches the closest available technician using **Uber-style real-time telemetry**.

---

## ✨ Core Features

- 🎙️ **Voice-Activated AI Triage**: Homeowners report emergencies simply by speaking. Azure Speech-to-Text captures the audio, and Azure OpenAI parses the text to determine the category (e.g., Plumbing) and urgency (e.g., High).
- ⚡ **Real-Time Uber-Style Dispatch**: Real-time Socket.IO communication instantly alerts the closest available technician.
- 🗺️ **Live En-Route Tracking**: Once a technician accepts a job, the homeowner sees their live vehicle location moving on a map (with dynamic simulated telemetry) along with ETA updates.
- 🔐 **Secure Arrival Verification**: The homeowner receives a 4-digit OTP. The technician must enter this OTP upon arrival to verify physical presence before mitigation begins.
- 💳 **Integrated Billing**: Simulated Razorpay checkout flow upon job completion, with automatic 80/20 platform commission splitting.

## 🛠️ Technology Stack

**Frontend (Client)**
- **Framework**: Next.js 14 (React 19)
- **Styling**: Tailwind CSS, PostCSS
- **Maps**: Leaflet / React-Leaflet
- **Voice**: Microsoft Cognitive Services Speech SDK
- **Real-Time**: Socket.IO Client

**Backend (Server)**
- **Framework**: Node.js with Express
- **Real-Time Engine**: Socket.IO
- **Database**: MongoDB (Mongoose ODN) — Hosted on MongoDB Atlas
- **AI Integration**: `@azure/openai`

**Infrastructure (Cloud Ready)**
- **Orchestration**: Kubernetes (Azure AKS)
- **Networking**: Azure Application Gateway (WAF v2) replacing Ingress
- **Containers**: Docker (Multi-stage builds)

---

## 🏗️ Architecture Design

CogniDispatch operates on a decoupled architecture designed for high availability on Kubernetes:

1. **Client-Side Voice Processing**: To protect backend bandwidth, audio is streamed directly from the browser to Azure Speech using temporary auth tokens generated by the backend (`/api/speech-token`).
2. **Stateful Telemetry**: The backend maintains live `Map` objects of active dispatches, emitting 3-second coordinate updates via Socket.IO rooms (`room_disp_<id>`).
3. **Stateless API**: All persistent data (Users, Vendors, Historical Dispatches) is saved to MongoDB, ensuring no data loss during Kubernetes Pod restarts.

## 🚀 Running Locally

### 1. Prerequisites
- Node.js (v18+)
- MongoDB Atlas cluster (Connection string)
- Azure Speech Resource Key (Central India or your preferred region)
- Azure OpenAI Endpoint & Key

### 2. Backend Setup
```bash
cd server
npm install

# Create your .env file
cp .env.example .env

# Start the server (runs on port 5000)
npm run dev
```

**Required `.env` variables (Backend):**
```env
PORT=5000
MONGODB_URI=mongodb+srv://<user>:<password>@<cluster>.mongodb.net/cognidispatch
AZURE_SPEECH_KEY=your_azure_speech_key
AZURE_SPEECH_REGION=centralindia
AZURE_OPENAI_ENDPOINT=https://your-openai.openai.azure.com/
AZURE_OPENAI_KEY=your_openai_key
AZURE_OPENAI_DEPLOYMENT=gpt-4o
ALLOWED_CLIENT_IP=http://localhost:3000
```

### 3. Frontend Setup
```bash
cd client
npm install

# Start the Next.js frontend (runs on port 3000)
npm run dev
```

### 4. How to Test the Flow
1. Open **[http://localhost:3000/technician](http://localhost:3000/technician)** in one browser tab. Log in as a Technician (e.g., `9090909090` / `password123`).
2. Open **[http://localhost:3000/homeowner](http://localhost:3000/homeowner)** in another tab. Log in as a Homeowner (e.g., `user1@gmail.com` / `password123`).
3. On the Homeowner tab, click the **Microphone** and say: *"My pipe burst and water is flooding everywhere!"*
4. Watch the AI parse the emergency and send an immediate pop-up offer to the Technician tab.
5. Accept the job on the Technician tab and watch the simulated vehicle drive to the homeowner's house in real-time.

## ☁️ Azure Deployment (AKS & Application Gateway)

The project includes production-ready Dockerfiles and Kubernetes manifests in the `/k8s` directory.

The deployment topology strictly utilizes **Azure Application Gateway (WAF v2)** instead of standard Nginx Ingress. The App Gateway directly routes traffic to NodePort services on the AKS nodes:
- `/api/*` and `/socket.io/*` ➔ Backend NodePort `30500`
- `/*` (default) ➔ Frontend NodePort `30080`

**Key Deployment Rule**: The frontend Docker image must be built with the public IP of the Application Gateway baked in as an argument:
```bash
docker build --build-arg NEXT_PUBLIC_SERVER_IP=http://<APPGW_PUBLIC_IP> -t cogni-frontend .
```

*For complete Azure CLI setup commands (VNet, Subnets, NSGs, NAT Gateway, AKS, AppGW), refer to the `azure_infra_guide.md` documentation in the artifacts.*
